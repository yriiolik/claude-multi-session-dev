# 跨模块协同编排（多模块接口交互的阶段化拆分：契约先行 / 提供方先行）

> 目的：当一个业务需求把多个模块用**接口/共享数据结构**串起来（提供方—消费方）时，避免「契约没定
> 就一起并行派，消费方按猜的接口写 → 返工 + 冲突」。改成**先定契约、再并行实现、最后独立联调**
> （模式 A），或**提供方先完整落地、消费方再按真实接口接入**（模式 B）。
> 设计依据业界成熟实践（见文末「业界依据」）。
> 本文件给主 session 用——主 session 负责**识别契约关系 + 选协同模式 + 评审契约**，**不写契约内容**
> （契约由 API 提供方 session 写）。
> ⚠ 不论哪种模式，提供方/消费方**各是独立子 session（一 session = 一模块，SKILL.md「派发粒度铁律」）**，
> 绝不因"反正有依赖"把两个模块合给一个 session。

## 一、何时触发 / 何时不需要（判据）

拆解第④问的落地判据。命中**任一**即走契约先行：

- 某模块要**调用**另一模块的**新接口**（HTTP / RPC / 内部 service 方法）；
- 多个模块要**读写同一个新的数据结构 / DTO / 事件 schema**，字段语义必须对齐；
- 提供方与消费方**分属不同 session**（要并行开发），且接口形状还没固定。

**不需要**契约先行（直接并行派、跳过段①）：

- 纯展示型：各端只是各自把已有/同一字段渲染出来，互不调用对方新接口；
- 各改各的：模块间无新接口/新共享结构耦合；
- 接口已稳定存在、本次不改其形状（消费方直接照现有契约接，提供方无改动）。

> 拿不准 → 当作「需要协同编排」处理：多一个契约设计 session 的成本，远小于消费方按错接口写完再返工。

## 一·五、选协同模式：模式 A 契约先行（并行） vs 模式 B 提供方先行（串行）

命中接口交互后，主 session 先在两种模式里选一种（默认模式 A）：

- **模式 A · 契约先行（默认，并行抢墙钟）**：提供方先出契约 → 主 session 评审定稿 → 提供方实现 +
  消费方按契约 mock 接入**并行** → 独立联调去 mock 拼真。下文「二、三段式」即模式 A 全流程。
- **模式 B · 提供方先行（串行，等真实接口）**：先派提供方**一张卡完整设计 + 开发 + 自测**（真实 API
  落地 + L2 文档 + 回执，契约文件仍建议落 `<COORD>/contracts/` 供消费方引用）；done 后主 session 从
  回执/契约提取**实际接口形态**写进消费方任务卡的「上下游协作 · 上游依赖」，**再派**消费方接入——
  消费方自测直接打真接口、无需 mock，**联调层通常可省**（自测已是真实拼接），验收层照旧。

**判据（命中任一倾向模式 B；都不命中 → 模式 A）：**
- 接口形状**强依赖实现探索**（要边做边定字段/口径），预先定稿大概率被实现推翻、契约反复改；
- **消费方接入量很小**（如只加一处调用/一列展示），等提供方做完的串行成本低于"写 mock + 独立联调"
  的开销；
- 不抢墙钟：串行总时长可接受，宁可少一层 mock 失真风险。

反之：双方工作量都大、接口可先于实现敲定、要并行抢墙钟 → 模式 A。

## 二、三段式（模式 A 全流程）

```
段①  契约设计（串行卡点，只派 提供方 1 个 session）
        提供方产出契约文件 → 主 session 评审定稿
                    │ 定稿
                    ▼
段②  分头开发（契约定稿后并行）
        提供方实现真逻辑        消费方按契约接入 + 对端 mock 自测
        （各自跑单测/模块 e2e，代码互不冲突）
                    │ 都 done
                    ▼
段③  联调（独立 联调 session，去 mock、真接口拼起来跑通）
```

### 段① 契约设计（只派提供方）

- 派发：`cc-dispatch --name "↳<provider>-contract@<RQ>" --env FLEET_ROLE=worker ...`
- 任务卡/preamble 注明：**本轮只产出契约、不实现业务逻辑**；契约文件落 `<COORD>/contracts/`。
- 提供方产出的**契约文件**应覆盖（用消费方能照着写桩的粒度）：
  - **接口签名**：方法/路由 + HTTP 动词 + 路径；
  - **请求 schema**：参数名、类型、必填性、**单位/口径**（如「数量」是套还是片——见项目约定）；
  - **响应 schema**：字段名、类型、嵌套结构、空值语义；
  - **错误码 / 失败语义**：每种失败返回什么码 + 业务含义；
  - **事件 / 副作用**（如有）：触发时机、payload schema；
  - **示例**：至少 1 个成功 + 1 个典型失败的 request/response 样例（消费方据此写 mock）。
- **主 session 评审契约（业务面，不读实现代码）**：字段齐不齐、口径/单位对不对、错误码覆盖了验收
  清单里的失败场景没、是否和 L1 业务需求一致。有问题 → 让提供方改契约再评审；OK → **定稿**。
- 定稿标记：把定稿版契约留在 `<COORD>/contracts/<name>.contract.md`（或 `.openapi.yaml` / `.ts` 类型
  文件，按项目技术栈），后续所有 session 引用同一份。

### 段② 分头开发（契约定稿后并行）

- **提供方**：按定稿契约实现真逻辑，自测覆盖契约里每条成功/失败语义。
- **消费方**：把契约文件作为**依赖锚点**写进任务卡。消费方**不等提供方做完**——按契约自建 mock/桩
  （用段①里的示例 request/response）接入并自测。双方代码互不冲突，真正并行。
- 任一方若发现契约**需要改**（实现中暴露契约缺漏/口径错）→ **不要各自偷偷改**，写进回执「需主
  session 裁决」，由主 session 改契约并通知相关 session 同步（契约是共享真相，单边改 = 又对不上）。

### 段③ 联调（独立 session）

- 段②所有相关模块 `done` 后，派**独立联调 session**：`--name "↳integ@<RQ>" --env FLEET_MODULE=integ`。
- 范围：**只读各模块 + 把它们真实拼起来（去 mock、真接口）跑端到端 / 集成测试**，**不改业务代码**。
- 回报：集成是否通、哪条接口对不上（请求/响应不匹配、口径错、错误码没按契约）、现象指向提供方还是
  消费方。回执写 `<COORD>/integ.summary.md`。
- 联调属「测试」不属「开发」——见「三、与三层测试的关系」。

## 三、与三层测试的关系

| 层 | 测什么 | 谁做 | 对端 |
|---|---|---|---|
| **自测** | 本模块改动相关的单测 / 模块 e2e | 各模块 worker（自己） | 按契约 **mock** |
| **联调（integration）** | 多模块真实拼接、接口对不对得上 | **独立 联调 session（↳integ）** | **真接口**（去 mock） |
| **验收（acceptance e2e）** | 端到端是否达成业务需求场景 | **独立 验收 session（↳verify）** | 真实全链路 |

- **场景由谁定**：自测场景=worker 自定（覆盖自己的契约语义）；**联调 + 验收场景=主 session 从验收
  口径设计**（测哪些场景由主 session 定），独立 session 只**执行**。
- 没走契约先行（无跨模块接口交互）的需求：通常没有「联调」层，自测 + 验收两层即可。

## 四、与三层文档（doc-traceability）的关系

契约是**跨模块共享的接口真相**，位置在 L1.5 委托 与各模块 L2 之间：

```
L1 业务需求 ──▶ L1.5 任务卡（委托）──▶ 契约文件（<COORD>/contracts/，跨模块共享）
                                            │
                          ┌─────────────────┼─────────────────┐
                          ▼                                     ▼
                  提供方 L2（实现契约）                 消费方 L2（按契约接入）
```

- 契约文件是**多模块都向下引用**的同一份；提供方 L2 说明「我如何实现这份契约」，消费方 L2 说明
  「我如何按这份契约接入」。两边 L2 的「↓下联」都指向同一个契约锚点。
- 契约由 **API 提供方 session 写**（它最懂自己暴露什么），**主 session 只评审、不代写**——同
  doc-traceability 里「主 session 不代写 L2」的理由（代写必 stale、违反 ownership、制造瓶颈）。

## 五、契约文件模板（提供方 session 填，主 session 评审）

> 路径：`<COORD>/contracts/<provider-or-capability>.contract.md`。技术栈支持时优先用机器可校验格式
> （OpenAPI `*.yaml` / TS 类型 `*.ts` / JSON Schema）并在本 md 里指向它。与代码同仓演化。

```markdown
# 契约: <能力/接口名>   (RQ: <RQ>  Provider: <provider-module>)

> Owner: 提供方 session（主 session 评审定稿，不代写）
> 上游业务需求(L1): docs/requirements/<NN>/README.md#<anchor>
> 消费方: <consumer-module-1>, <consumer-module-2>

## 接口
- 名称 / 路由: <METHOD> /api/.../<...>   （或内部 service: <Class>.<method>）
- 用途（业务语言）: <这个接口为支撑哪条业务需求而存在>

## 请求 schema
| 字段 | 类型 | 必填 | 口径/单位 | 说明 |
|---|---|---|---|---|
| ... | ... | ✓ | 套/片/… | ... |

## 响应 schema
| 字段 | 类型 | 空值语义 | 说明 |
|---|---|---|---|
| ... | ... | ... | ... |

## 错误码 / 失败语义
| 码 | 触发条件 | 业务含义 |
|---|---|---|
| 409 INSUFFICIENT_STOCK | 可用量 < 需求量 | 库存不足，下单被拦 |

## 事件 / 副作用（如有）
- <事件名>: 触发时机 + payload schema

## 示例（消费方据此写 mock）
- 成功: <request> → <response>
- 失败: <request> → <error response>

## 变更记录
- v1 (定稿): ...
- v2: <谁、为何改、影响哪些消费方——契约改动必须主 session 裁决后统一同步>
```

## 业界依据（来源）

- **API-first / Contract-first**：先定接口契约（OpenAPI / Protobuf）再并行实现，前后端/服务间解耦
  并行开发。 [OpenAPI Initiative](https://www.openapis.org/) ·
  [Swagger: API-First Design](https://swagger.io/resources/articles/adopting-an-api-first-approach/)
- **Consumer-Driven Contracts（CDC）**：消费方按契约写测试、提供方据此校验，避免集成期才发现接口
  对不上。 [Fowler: Consumer-Driven Contracts](https://martinfowler.com/articles/consumerDrivenContracts.html) ·
  [Pact](https://docs.pact.io/)
- **Contract testing vs integration**：契约测试在各侧独立跑（mock 对端），真集成（去 mock）单列一层
  ——对应本文「自测 / 联调」分层。 [Fowler: Integration Test](https://martinfowler.com/bliki/IntegrationTest.html)
- **Stub/mock 解耦并行**：消费方用桩按契约自测、不等提供方做完，是契约先行能并行的关键。
  [Fowler: TestDouble](https://martinfowler.com/bliki/TestDouble.html)
